## 1. NanoChat vs Parameter Golf — Dataset & Benchmark Alignment

The central discussion point: nanochat and Parameter Golf are two parallel tracks in this project, but they use different datasets and different benchmarks. This makes direct comparison impossible without alignment work. The team agreed that for the Parameter Golf track, we need nanochat to consume the exact same data and report the same metric as the competition baseline.

### What's different

| Aspect | nanochat (default) | Parameter Golf |
|---|---|---|
| Dataset | FineWeb-EDU / ClimbMix-400B | FineWeb sp1024 |
| Vocab size | 32,768–65,536 | 1,024 |
| Tokenizer | Custom Rust BPE (trained from scratch) | SentencePiece BPE (`fineweb_1024_bpe.model`) |
| Primary benchmark | CORE (22-task ensemble: ARC, HellaSwag, MMLU, TriviaQA, etc.) | BPB (bits per byte) on FineWeb validation |
| Training budget | ~4 hours on 8xH100 | 10 minutes on 8xH100 |
| Artifact constraint | None | 16 MB (code + compressed weights) |

Both use FineWeb as the underlying data source, but Parameter Golf retokenizes it with a tiny 1024-token SentencePiece vocabulary and measures only BPB (a tokenizer-agnostic compression metric). nanochat's default setup uses a much larger vocabulary and evaluates on a broad set of downstream tasks via CORE.

### How we bridge them

nanochat already supports flags for consuming pretokenized data:

```bash
--pretokenized-data-dir "$PG_DATA_DIR"   # point at Parameter Golf's .bin shards
--vocab-size 1024                         # match the sp1024 tokenizer
--tokenizer-path "$PG_TOKENIZER"          # use Parameter Golf's SentencePiece model
```

The script `runs/nanochat_vs_pgolf.sh` automates an apples-to-apples comparison: it runs both Parameter Golf's `train_gpt.py` and nanochat's `base_train.py` on the identical FineWeb sp1024 data, extracts the final BPB from each, and prints a side-by-side comparison.

### Decision

- **Parameter Golf experiments**: standardize on BPB as the sole metric, using FineWeb sp1024 data
- **CS 7643 course project**: keep CORE as the primary benchmark for the learning-curve comparison
- Both tracks can share infrastructure (RunPod pods, scripts, wandb dashboard)

## 2. BPB Compression & Quantization

A team member raised the topic of model compression and its implications for the 16 MB artifact budget.

### Compression results from our first run

| Metric | Value |
|---|---|
| Raw model size (bf16) | 67.2 MB |
| Compressed (int8 + zlib) | 14.7 MB |
| Compression ratio | 3.91x |
| BPB before compression | 1.3033 |
| BPB after roundtrip | 1.3045 |
| BPB degradation | +0.001 (negligible) |
| Under 16 MB budget? | Yes (1.3 MB headroom) |

### Why this matters

The 16 MB artifact budget is the hardest constraint in Parameter Golf. The fact that int8+zlib achieves nearly lossless compression (only +0.001 BPB degradation) means we can **decouple** two problems:

1. **Train the best model** — optimize architecture, data, and schedule purely for lowest BPB
2. **Fit in 16 MB** — apply int8+zlib post-training, which compresses ~3.9x with negligible quality loss

This means we can train models up to ~62 MB raw (bf16) and still fit under budget after compression.

### FP8 training

nanochat supports `--fp8` for FP8 precision training on H100+ hardware. This gives ~17% raw training throughput improvement. In a 10-minute wall-clock competition, 17% more throughput means more training steps and lower BPB. FP8 training and int8+zlib compression are independent techniques — FP8 optimizes training speed, int8+zlib optimizes artifact size. They compose.

## 3. What is CLAUDE.md and SKILL.md

For teammates who haven't used AI coding assistants before, here's what these files are and why we use them.

### CLAUDE.md — persistent project instructions

CLAUDE.md is a Markdown file at the root of a repository that provides persistent instructions to **Claude Code** (Anthropic's AI coding CLI, `claude.ai/code`). When Claude Code opens a project, it automatically reads CLAUDE.md and uses it as context for all interactions in that session.

Think of it as a README, but for the AI — it tells Claude Code what the project is about, what conventions to follow, what the current state is, and what decisions have been made. Without it, the AI starts from scratch every session.

**What ours contains:**
- Project overview (nanochat learning-curve comparison + Parameter Golf)
- Repository structure map
- Framework comparison context (nanoGPT vs llm.c vs nanochat)
- Context history — dated log of what was done, found, and decided across sessions

There are two levels:

| File | Scope | Location | In git? |
|---|---|---|---|
| Project CLAUDE.md | Per-repository instructions | `<repo>/CLAUDE.md` | Yes — shared with the team |
| Global CLAUDE.md | Personal preferences (all projects) | `~/.claude/CLAUDE.md` | No — private to each person |

The project-level CLAUDE.md is committed to git, so when any team member uses Claude Code on this repo, the AI already "knows" the project context. The global one contains personal preferences (communication style, coding conventions) and stays private.

### SKILL.md — reusable skills (plugins)

SKILL.md files define reusable capabilities for AI coding tools — like plugins. Each skill lives in its own directory under `.claude/skills/` and contains:

- **Metadata** — file patterns and bash patterns that trigger the skill automatically
- **Instructions** — what the AI should do when the skill activates
- **Priority** — which skills take precedence when multiple match

For example, a skill might activate whenever you edit a `.tsx` file and inject React best-practice guidelines, or activate on `git push` and run a pre-push checklist. Skills extend what the AI can do without you having to repeat instructions every session.

### Equivalent files in other AI tools

The concept of "give the AI persistent project context" exists across all major AI coding tools:

| Tool | Project instructions | Skills / plugins |
|---|---|---|
| **Claude Code** | `CLAUDE.md` | `.claude/skills/*/SKILL.md` |
| **OpenAI Codex CLI** | `AGENTS.md` | Custom agents |
| **Cursor** | `.cursorrules` | — |
| **GitHub Copilot** | `.github/copilot-instructions.md` | — |
| **Windsurf** | `.windsurfrules` | — |

We're using CLAUDE.md because the team is standardized on Claude Code. The file is shared via git so everyone benefits from the accumulated project context.
