P1a vs P3 Comparison — Post-Retraction Analysis
=================================================
Summary: P1c achieved 0.9978 BPB (sub-1.0!) but has THREE bugs: gate OFF, CaseOps byte
inflation, and CaseOps training data via symlinks. Corrected BPB ≈ 1.087 (worse than C6).
P1a (1.0769) is best BPB without bugs. C6 (1.0805) remains best legal submittable run.

Run Descriptions:
  C6:     PR #1851 base + headwise gate + emb7+eclip15. Default batch/EMA/schedule.
  P1a:    C6 + 6 SOTA hparam overrides (warmdown, min_lr, beta2, clip sigmas, gptq_reserve).
          Same default batch size. Zero code changes vs C6, env vars only. OVER BUDGET.
  P1c:    P1a but MATRIX_CLIP_SIGMAS tightened (12.0 vs 11.5) to fit under 16 MB.
          Hypothesis: tighter clip costs ~0.001 BPB but saves ~0.4 MB.
  P3-fix: PR #1851 base + headwise gate + EMA=0.990 + small batch (196K) + emb6.
          CaseOps data + CASEOPS_ENABLED=1 (sidecar byte counting).
          4× smaller batch → ~3× more steps, but noisier gradients.


● Key findings from the log analysis:

  The byte ratio is WRONG again. Computing BPB × ln(2) / val_loss:
  - P4: 1.00741 × 0.6931 / 2.8511 = **0.2449**
  - C6 (standard): 0.2683

  That's ~9.6% inflation — the same CaseOps-like deflation we saw in P3/P1c. Despite removing the symlink on the data directory, the tokenizer might still be
   CaseOps, or PR #1851's code computes bytes differently.

  The val_loss tells the real story:
  - P4 val_loss: 2.8535 (worse)
  - C6 val_loss: ~2.8092 (better)

  Normalized to C6's byte ratio: P4 true BPB ≈ 1.104 — worse than C6 (1.0805) by +0.024.

  Why worse? P4 only got 2,542 steps vs C6's ~4,467. The layer loop activation at step 1442 cut throughput by ~45% (8.2M → 3.8M tok/s), so P4 saw far fewer
  tokens in the same 10-minute window.

  Action items:
  1. Check if the tokenizer is still CaseOps: ls -la data/ — look for fineweb_8192_bpe.model symlink
  2. Install python-minifier and rerun — but the result will likely be worse than C6
  3. The deeper issue: SOTA hparams (WARMDOWN=0.85, MIN_LR=0.10) may need more steps to pay off, and the throughput loss from layer loop prevents that

────────────────────────────────────────────────────────────────────────────────
2a. P1c Bugs Found
────────────────────────────────────────────────────────────────────────────────

Bug 1: GATED_ATTN env var mismatch — gate was OFF
  - p4_clip12.env had `GATED_ATTN=headwise` (V2 train_gpt_v2.py style)
  - PR #1851 train_gpt.py reads `GATED_ATTN_ENABLED` (boolean, 0/1)
  - Result: gated_attn_enabled: False in logs. Gate was completely OFF.
  - The 0.9978 BPB was achieved WITHOUT gated attention.
  - Fix: add `export GATED_ATTN_ENABLED=1` to config.

Bug 2: CaseOps byte accounting — inflated byte denominator by ~9%
  - Pod had CaseOps symlinks: fineweb10B_sp8192 → fineweb10B_sp8192_lossless_caps_caseops_v1_reserved
  - CASEOPS_ENABLED was not set → code used LUT byte counting on CaseOps tokenizer
  - LUT includes case marker bytes not present in raw text → inflated denominator
  - Token/byte ratio proof:
      C6 (standard):    BPB×ln2/val_loss = 1.08753×0.6931/2.80919 = 0.2683
      P1c (bugged LUT): BPB×ln2/val_loss = 1.00290×0.6931/2.39123 = 0.2907  ← neither standard nor correct
      P3 (sidecar):     BPB×ln2/val_loss = 1.09260×0.6931/2.39117 = 0.3167  ← correct CaseOps
  - P1c ratio (0.2907) falls between standard and correct — exactly where LUT inflation lands
  - Correction factor: 0.3167 / 0.2907 = 1.0894 (~8.9% inflation, matches P3's ~9%)
  - Corrected P1c BPB: Pre-Q 1.093 | Quant 1.100 | TTT **1.087** (worse than C6 by +0.0065)

Bug 3: CaseOps training data — model trained on wrong data
  - Pod symlinks pointed standard SP8192 paths to CaseOps-tokenized data
  - Model was trained on CaseOps tokens, NOT standard SP8192
  - val_loss (2.3912) reflects CaseOps data quality, not standard data
  - Even with correct byte counting, results are not comparable to C6/P1a (different data)
  - A clean re-run needs: fresh pod with standard data download (no symlinks)

Bug 4: Still over budget (16.69 MB) even without gate
  - MATRIX_CLIP_SIGMAS=12.0 not tight enough.
  - With gate ON, model will have MORE parameters → even larger file.
  - Fix: tighten MATRIX_CLIP_SIGMAS to 12.5 for re-run.

Additional observations:
  - model_params: 35,944,602 (without gate — gate adds ~headwise dim params)
  - Pre-Q BPB: 1.0029 | Quant BPB: 1.0097 | TTT BPB: 0.9978 (ALL INFLATED — see Bug 2)
  - Eval time: 443.6s | Seed: 1337
  - The sub-1.0 BPB was an artifact. Corrected TTT BPB ≈ 1.087.

────────────────────────────────────────────────────────────────────────────────
3. P3 Byte Accounting Error (Root Cause)
────────────────────────────────────────────────────────────────────────────────

Causal chain:
  1. Pod setup created symlinks: standard paths → CaseOps data + tokenizer
       ln -s fineweb10B_sp8192_lossless_caps_caseops_v1_reserved → fineweb10B_sp8192
       ln -s fineweb_8192_bpe_lossless_caps_caseops_v1_reserved.model → fineweb_8192_bpe.model
  2. Config set CASEOPS_ENABLED=0 (assumed irrelevant since data was already tokenized)
  3. Code has two byte-counting paths:
       CASEOPS_ENABLED=1 → loads sidecar file (fineweb_val_bytes_*.bin) = correct raw bytes
       CASEOPS_ENABLED=0 → calls build_sentencepiece_luts() = bytes from tokenizer vocab
  4. LUT path computed len(piece.encode("utf-8")) on CaseOps tokenizer pieces.
     CaseOps pieces include case marker characters (not present in raw text).
  5. Result: byte denominator ~164.6M (inflated) vs ~151M (canonical) = +8.3%
     BPB formula: loss/ln2 × tokens/bytes → smaller denominator = higher BPB
     Reported 1.0066 → actual 1.0975 (2-seed mean with correct sidecar counting)

Fix: CASEOPS_ENABLED=1 + explicit data/tokenizer paths (no symlinks) + sidecar file.
The val_loss (~2.40) was always genuine — only the BPB denominator was wrong.

4. Key Takeaways
────────────────────────────────────────────────────────────────────────────────

1. BEST LEGAL RUN: C6 at 1.0805 BPB (3-seed mean, under budget, submitted PR #2005).
   P1a beats it by −0.0036 but is 0.35 MB over budget.

2. P3's techniques FAILED at 8×H100 scale (2-seed confirmed):
   - EMA=0.990: proved on 2×H100 (−0.012) but HURT on 8×H100 (+0.0025 in L1).
     Even with 12,140 steps (small batch), aggressive EMA doesn't help.
   - Small batch: more steps (12,140 vs 4,467) but smaller batches degrade quality
     more than extra updates help. L2 confirmed: +0.0121 vs C6.
   - Combined (P3-fix): +0.0170 vs C6 (2-seed mean). Techniques compound negatively.

3. P1a proves SOTA hparams transfer cleanly: zero code changes, just 6 env var
   overrides from PR #1855. WARMDOWN=0.85 + MIN_LR=0.10 + BETA2=0.99 are the
   likely drivers (extended training via warmdown + non-zero floor LR).

4. P1a budget problem is MATRIX_CLIP_SIGMAS=11.5 (looser than C6's 12.85).
   Allows larger weight magnitudes → compresses worse under brotli.
   Fix: tighten to 12.0-12.5 to fit under 16 MB while retaining most gain.

5. 2×H100 findings don't transfer to 8×H100:
   - EMA=0.990: −0.012 on 2×H100 → +0.003 on 8×H100
   - Small batch: −0.015 on 2×H100 → +0.012 on 8×H100
   Root cause: 8 GPUs already give sufficient batch diversity; reducing batch
   per-GPU just hurts signal-to-noise without the step-count payoff.

6. val_loss vs BPB disconnect: P3-fix has LOWER val_loss (2.4012 vs 2.7816)
   but HIGHER BPB (1.0972 vs 1.0769). This is because BPB = loss/ln2 × tokens/bytes,
   and the token/byte ratio differs between CaseOps and standard tokenization.

7. Path forward for beating C6 legally:
   - P1c re-run on CLEAN pod (no CaseOps symlinks) with:
     GATED_ATTN_ENABLED=1 + MATRIX_CLIP_SIGMAS=12.5 + standard SP8192 data
   - P1c's corrected BPB (1.087) is WORSE than C6 (1.0805) — sub-1.0 was an artifact
   - SOTA hparams still likely help (P1a = 1.0769 pre-bug), but gains are modest
   - Or: cherry-pick individual SOTA hparams (WARMDOWN, MIN_LR) that don't
     affect compression, skip MATRIX_CLIP_SIGMAS change entirely

9. P1c has THREE bugs (gate OFF + byte inflation + wrong training data).
   Only bug #1 (gate) was caught initially. Bugs #2-3 from pod CaseOps symlinks
   are the same root cause as P3's retraction. Any run on a pod with CaseOps
   symlinks is suspect — always verify token/byte ratio against C6 baseline (0.2683).

8. PreQuantTTT is ruled out (C3 violation) — cannot use the technique that
   made P3's val_loss so good (2.40 vs 2.78).
